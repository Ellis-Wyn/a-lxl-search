## 一、项目概述

### 1.1 项目定位

**病理AI药研情报库** 是一个专注于药物研发第一手资讯的智能数据库系统，帮助研发人员快速掌握：
- 进入临床 I/II/III 期的靶点与分子
- CDE 受理/审评相关公开信息
- 药企管线竞争格局
- PubMed 最新文献进展

### 1.2 核心价值主张

| 维度 | 价值 |
|------|------|
| **数据时效性** | 每日凌晨2点自动爬取最新数据 |
| **智能搜索** | 支持公司名称别名、中英文名称、简称搜索 |
| **数据归一化** | 自动标准化研发阶段、适应症、公司名称 |
| **来源追溯** | 每条数据都可追溯到原始来源 URL |
| **竞品监控** | 自动检测 Phase Jump 和竞品退场 |

### 1.3 当前数据规模（基于代码分析）

| 数据类型 | 预估数量 | 来源 |
|---------|---------|------|
| 研发管线 | 167+ 条 | 12-14家药企官网 |
| 靶点信息 | 动态增长 | 数据库存储 + 自动提取 |
| PubMed文献 | 实时搜索 | NCBI Entrez API |
| 药企覆盖 | 14家公司 | 官网爬虫 |

## 三、数据库结构详解

### 3.1 核心数据模型（5张主表 + 2张关联表）

#### 3.1.1 Pipeline（管线表）- 核心表

**文件位置**: `models/pipeline.py`

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| pipeline_id | UUID | 主键 | ✓ |
| drug_code | VARCHAR(255) | 药物代号/名称 | ✓ |
| company_name | VARCHAR(255) | 公司名称 | ✓ |
| indication | TEXT | 适应症 | - |
| phase | VARCHAR(50) | 研发阶段（标准化） | ✓ |
| phase_raw | VARCHAR(255) | 原始阶段名称 | - |
| modality | VARCHAR(100) | 药物类型 | ✓ |
| source_url | TEXT | 原始来源URL | - |
| first_seen_at | TIMESTAMP | 首次发现时间 | ✓ |
| last_seen_at | TIMESTAMP | 最近更新时间 | ✓ |
| status | VARCHAR(50) | 状态 | ✓ |
| discontinued_at | TIMESTAMP | 终止时间 | - |
| is_combination | BOOLEAN | 是否联合用药 | ✓ |
| combination_drugs | TEXT | 联合用药列表(JSON) | - |

**唯一约束**: `(drug_code, company_name, indication)`

**核心方法**:
- `has_phase_changed()`: 检测阶段变化
- `is_disappeared(threshold_days=21)`: 检测竞品退场
- `get_phase_order()`: 获取阶段排序值

#### 3.1.2 Target（靶点表）- 核心锚点

**文件位置**: `models/target.py`

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| target_id | UUID | 主键 | - |
| standard_name | VARCHAR(100) | 标准名称（唯一） | ✓ |
| aliases | JSONB | 别名数组 | - |
| gene_id | VARCHAR(50) | NCBI Gene ID | ✓ |
| uniprot_id | VARCHAR(50) | UniProt ID | ✓ |
| category | VARCHAR(100) | 靶点分类 | - |
| description | TEXT | 靶点描述 | - |

**核心特性**:
- 支持通过别名反向查找（GIN索引）
- 一个靶点可关联多篇文献和多个管线

#### 3.1.3 Publication（文献表）- 科学证据

**文件位置**: `models/publication.py`

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| pmid | VARCHAR(20) | PubMed唯一标识（主键） | - |
| title | TEXT | 文献标题 | - |
| abstract | TEXT | 摘要全文 | - |
| pub_date | DATE | 发布日期 | ✓ |
| journal | VARCHAR(255) | 期刊名 | ✓ |
| source_type | VARCHAR(100) | 来源类型 | ✓ |
| mesh_terms | JSONB | MeSH主题词数组 | - |
| clinical_data_tags | JSONB | 临床数据指标 | - |
| publication_type | VARCHAR(50) | 文献类型 | ✓ |

**核心方法**:
- `calculate_recency_score(max_days=730)`: 计算时效性得分
- `has_clinical_data()`: 是否包含临床数据
- `get_clinical_metrics()`: 提取临床指标（ORR/PFS/OS）

#### 3.1.4 CDEEvent（CDE事件表）- 监管信息

**文件位置**: `models/cde_event.py`

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| id | VARCHAR(100) | 主键（受理号） | - |
| event_type | VARCHAR(50) | 事件类型(IND/CTA/NDA/BLA) | ✓ |
| acceptance_no | VARCHAR(100) | 受理号（唯一） | ✓ |
| drug_name | VARCHAR(200) | 药品名称 | ✓ |
| applicant | VARCHAR(200) | 申请人/企业名称 | ✓ |
| undertake_date | DATE | 承办日期 | ✓ |
| indication | TEXT | 适应症描述 | - |
| public_page_url | VARCHAR(500) | CDE公示页面URL | - |
| source_urls | JSON | 所有相关URL数组 | - |

#### 3.1.5 关联表

**TargetPublication（靶点-文献关联）**:
- 联合主键: `(target_id, pmid)`
- relation_type: mentioned_in / focus_on
- evidence_snippet: 原文片段

**TargetPipeline（靶点-管线关联）**:
- 联合主键: `(target_id, pipeline_id)`
- relation_type: targets / inhibits / antibody_to / agonist_of
- is_primary: 是否主靶点（多靶点药物）

## 四、核心功能模块

### 4.1 统一搜索引擎

**文件位置**: `services/unified_search_service.py`, `api/search.py`

**功能特性**:
- 一次调用搜索所有实体类型（管线+文献+靶点+CDE事件）
- 智能查询扩展（同义词、全名自动匹配）
- 相关性评分排序
- 多维度筛选（公司、阶段、日期、事件类型）
- 自动缓存加速

**API端点**: `GET /api/search/unified`

**请求参数**:
```python
q: str                    # 搜索关键词（必需）
type: str                 # all/pipeline/publication/target/cde_event
company: str              # 公司名称筛选
phase: str                # 阶段筛选
moa_type: str             # MoA类型筛选
journal: str              # 期刊筛选
date_from/date_to: str    # 日期范围
event_type: str           # CDE事件类型
limit: int                # 结果数量限制
```

**响应格式**:
```json
{
  "query": "EGFR",
  "total_count": 170,
  "results": {
    "pipelines": {"count": 50, "items": [...]},
    "publications": {"count": 80, "items": [...]},
    "targets": {"count": 20, "items": [...]},
    "cde_events": {"count": 20, "items": [...]}
  },
  "facets": {
    "companies": {"恒瑞医药": 30, "百济神州": 20},
    "phases": {"Phase 3": 25, "Phase 2": 15}
  }
}
```

### 4.3 数据归一化服务

**文件位置**: `services/data_normalization_service.py`

**归一化类型**:

1. **Phase归一化**:
   - "临床I期" → "Phase 1"
   - "Phase II" → "Phase 2"
   - "已上市" → "Approved"
   - "申报中" → "Filing"

2. **公司名称归一化**:
   - 支持35家公司的别名映射
   - "恒瑞" → "恒瑞医药"
   - "百济" → "百济神州"

3. **适应症归一化**:
   - "非小细胞肺癌" → "NSCLC"
   - "三阴性乳腺癌" → "TNBC"

### 4.4 竞品监控与预警

**监控类型**:

1. **Phase Jump检测**:
   - 检测阶段前进（Phase II → Phase III）
   - 自动触发告警

2. **竞品退场预警**:
   - 连续21天未抓取到标记终止
   - 阈值可配置

3. **新入局者识别**:
   - 公司首次进入某靶点
   - 新管线自动识别

## 五、爬虫系统架构

### 5.1 爬虫基类

**文件位置**: `crawlers/base_spider.py`

**核心类**: `CompanySpiderBase`

**主要功能**:
- HTTP请求（带重试、缓存、熔断器）
- HTML解析（BeautifulSoup）
- 数据标准化（Phase Mapper）
- 数据库入库
- 性能监控
- 错误处理

**核心方法**:
```python
fetch_page(url, timeout, use_cache)    # 获取页面
parse_html(html)                        # 解析HTML
normalize_phase(raw_phase)              # 标准化阶段
save_to_database(item)                  # 保存到数据库
check_discontinued_pipelines(codes)     # 检测消失管线
discover_pipeline_url(homepage_url)     # 自动发现管线URL
```

### 5.2 爬虫示例：恒瑞医药

**文件位置**: `crawlers/hengrui_spider.py`

**爬取流程**:
1. 访问官网: `https://www.hengrui.com`
2. 自动发现管线页面URL
3. 解析管线列表HTML
4. 提取每条管线数据:
   - 药物代码（如 HRS-xxxx）
   - 适应症
   - 研发阶段
   - 靶点信息
5. 标准化数据
6. 入库
7. 检测消失管线

**数据结构**:
```python
PipelineDataItem(
    drug_code="HRS-1234",
    company_name="恒瑞医药",
    indication="非小细胞肺癌",
    phase="Phase 3",
    modality="单抗",
    source_url="https://...",
    targets=["EGFR"],
    is_combination=False
)
```

### 5.3 爬虫特性

1. **遵守robots.txt**:
   - 请求延迟: 0.3-0.5秒
   - 速率限制保护

2. **容错机制**:
   - 重试机制（指数退避）
   - 熔断器保护
   - 响应缓存

3. **性能监控**:
   - 请求成功率
   - 响应时间统计
   - 缓存命中率

**或者使用 Python 脚本备份**（创建 `scripts/backup_database.py`）：

```python
"""
数据库备份脚本
"""
import subprocess
import datetime
from pathlib import Path

def backup_database(output_dir="backups"):
    """备份数据库到 SQL 文件"""

    # 创建备份目录
    output_path = Path(output_dir)
    output_path.mkdir(exist_ok=True)

    # 生成备份文件名
    timestamp = datetime.datetime.now().strftime("%Y%m%d_%H%M%S")
    backup_file = output_path / f"drug_intelligence_db_{timestamp}.sql"

    # 执行备份
    cmd = [
        "pg_dump",
        "-h", "localhost",
        "-U", "postgres",
        "-d", "drug_intelligence_db",
        "--clean",  # 删除现有表再创建
        "--no-owner",  # 不导出所有者信息
        "--no-privileges",  # 不导出权限信息
        "-f", str(backup_file)
    ]

    print(f"正在备份数据库到: {backup_file}")
    subprocess.run(cmd, check=True)
    print(f"✓ 备份完成: {backup_file}")

    return backup_file

if __name__ == "__main__":
    backup_database()
```

## 四、自动化部署脚本

创建一键部署脚本 `scripts/deploy_to_server.sh`：

```bash
#!/bin/bash
# 一键部署脚本

set -e  # 遇到错误立即退出

# 配置变量
SERVER_USER="ubuntu"
SERVER_IP="your-server-ip"
PROJECT_DIR="/opt/pathology-ai"
BACKUP_FILE="backup.sql"

echo "========================================="
echo "病理AI药研情报库 - 一键部署脚本"
echo "========================================="

# 1. 备份本地数据库
echo "步骤 1: 备份本地数据库..."
pg_dump -h localhost -U postgres -d drug_intelligence_db \
  --clean --no-owner --no-privileges \
  -f $BACKUP_FILE
echo "✓ 备份完成: $BACKUP_FILE"

# 2. 上传代码到服务器
echo "步骤 2: 上传代码..."
rsync -av --exclude='__pycache__' \
  --exclude='*.pyc' \
  --exclude='logs/*' \
  --exclude='.env' \
  ../ $SERVER_USER@$SERVER_IP:$PROJECT_DIR/
echo "✓ 代码上传完成"

# 3. 上传备份文件
echo "步骤 3: 上传数据库备份..."
scp $BACKUP_FILE $SERVER_USER@$SERVER_IP:/tmp/
echo "✓ 备份文件上传完成"

# 4. 在服务器上启动服务
echo "步骤 4: 启动 Docker 服务..."
ssh $SERVER_USER@$SERVER_IP << EOF
  cd $PROJECT_DIR/back_end

  # 启动 Docker Compose
  docker-compose up -d

  # 等待数据库就绪
  sleep 10

  # 恢复数据
  docker cp /tmp/$BACKUP_FILE pathology-ai-postgres:/tmp/
  docker exec -i pathology-ai-postgres psql \
    -U pathology_user \
    -d pathology_db \
    < /tmp/$BACKUP_FILE

  echo "✓ 服务启动完成"
EOF

echo "========================================="
echo "部署完成！"
echo "访问地址: http://$SERVER_IP:8000"
echo "========================================="
```
